Bottom line: hf-sandbox is a functional Modal-style API on top of HF Jobs. It's fine for single-shot or small-fan-out workloads. It is not currently a drop-in replacement for E2B in any scenario that requires fast cold starts or ≥30 concurrent sandboxes — the Cloudflare-tunnel architecture caps the concurrent-create rate. For our existing eval sweeps (conc=100), E2B remains the right choice until hf-sandbox's tunnel stack is hardened.

Why: E2B uses snapshot-based templates that resume in ~1 s. hf-sandbox starts a fresh HF Job, installs a FastAPI RPC server, opens a Cloudflare tunnel, then becomes reachable — ~20-25 s end-to-end.

Two patterns:

  1. Small files are RPC-bound on both (single op ≈ 250-400 ms regardless of size <100 KB).
  2. Large reads on E2B exploit a streaming path (~34 MB/s) — HF tops out near 5 MB/s. Matters for pulling agent trajectories or trained-artifact downloads.

B04 — Concurrent create (the headline scalability number)

Fan out N parallel create() → exec("echo ready") → terminate() cycles.

NE2B successE2B wallHF successHF wallHF errors
55/5 (100%)2.1 s5/5 (100%)25.6 s—
2020/20 (100%)2.2 s20/20 (100%)46.0 s—
5050/50 (100%)2.1 s29/50 (58%)88.1 s21 tunnel

E2B is essentially flat — 50 concurrent boots in the same wall time as 5 (2.1 s). Templates are cached and the platform absorbs the fan-out.

HF cliffs around N ≈ 30-50 — Cloudflare-tunnel registration races / saturates, causing 21 of 50 sandboxes to die before they're reachable. Below N=20 it holds, but every successful boot still costs ~25-46 s wall-time.

Long-running interactive coding agentE2B — boot time (< 1 s vs 23 s)

HF-native pipelines (training jobs that need scratch sandboxes)hf-sandbox fine at N ≤ 20

Caveats

  1. hf-sandbox is v0.1.1 — the API surface (5 methods) is minimal vs E2B's richer one (snapshots, MCP, volume mounts, network policies). Different scope, not strictly an E2B competitor.
  2. Free trycloudflare tunnels are best-effort. Production HF-Jobs deployments would presumably swap in a stable reverse proxy; until then the concurrent-create ceiling we measured is the real one.
  3. HF Job quota — our account has Pro tier; free tier limits are tighter.
  4. Only CPU-basic measured. GPU-tier behavior may differ.
  5. Sample sizes are modest (n=5 for B01, n=10/20/50 for B04). Numbers are directional — the order-of-magnitude gaps in boot latency and N=50 failure rate are robust to that.
